Constraint Architecture

Rules can only tighten.
Never loosen.

ZeroPoint evaluates every action through a layered constraint hierarchy. Constitutional rules are hardcoded and non-removable. Operational rules compose additively. External systems can inject constraints but can never weaken existing ones. The most restrictive decision always wins.

For integrators building cognitive, compliance, or domain-specific layers on top of ZeroPoint.

Decision Spectrum

Five graduated responses to every action

Not everything is allow/deny. ZeroPoint returns graduated decisions ranked by severity. The engine collects decisions from all rules and returns the most restrictive. A single Block from any rule overrides Allow from every other rule.

Severity Decision Meaning
5 Block Action denied. No override possible for constitutional blocks.
4 Review Action suspended pending human approval. Configurable timeout.
3 Warn Action proceeds only after explicit acknowledgment.
2 Sanitize Action proceeds with modifications — redaction, scoping, parameter rewriting.
1 Allow Action proceeds unconditionally.
The Hierarchy

Three layers. Strict evaluation order.

Rules evaluate in a fixed order. Each layer can only add constraints, never remove them. Constitutional rules evaluate first and cannot be bypassed by any mechanism — native, WASM, or external.

Layer 1 — Constitutional

Hardcoded. Non-removable. Evaluated first.

These are ZeroPoint's tenets, enforced in compiled Rust. No capability grant, no policy rule, no WASM module, no external system can bypass them. There is no removal API. They exist in code.

HarmPrincipleRule — Tenet I: Do No Harm

Blocks actions whose targets or parameters indicate intent to cause harm: weaponization, surveillance of individuals, deepfakes, impersonation, exploitation, suppression of dissent. Inspects action targets and tool names.

SovereigntyRule — Tenet II: Sovereignty Is Sacred

Blocks configuration changes that would undermine sovereignty guarantees: disabling the Guard, bypassing capability checks, truncating the audit trail, forging capability grants, removing constitutional rules, overriding sovereign refusal.

Layer 2 — Operational

Configurable. Extensible. Cannot override Layer 1.

Native Rust rules and WASM policy modules compose here. New rules can be added at runtime. Existing rules cannot be removed. WASM modules execute in memory-isolated sandboxes — they cannot access the host filesystem, network, or other modules.

CatastrophicActionRule — Blocks credential exfiltration and recursive self-modification. Always.
BulkOperationRule — Warns on file operations matching glob or recursive patterns above a configurable threshold.
ReputationGateRule — Gates mesh actions by peer reputation. High-risk actions require Good standing. Unknown peers trigger human Review.
WASM Modules — Sandboxed, fuel-limited policy rules. Receive the full PolicyContext as JSON, return a PolicyDecision as JSON. Deployed at runtime without recompiling ZeroPoint.
Layer 3 — Baseline

Default permit. Evaluated last.

The DefaultAllowRule permits any action that no other rule has restricted. This represents ZeroPoint's "default open" posture. Can be replaced with a default-deny rule for strict environments.

Integration Boundary

What external constraints can and cannot do

External systems — cognitive layers, compliance engines, domain-specific policy providers — can inject constraints through native Rust rules or WASM modules. The boundary is strict.

Can do

Add new Block rules for domain-specific requirements
Add Review gates that suspend actions for human oversight
Add Warn thresholds for domain-specific risk patterns
Add Sanitize transforms that redact or scope action parameters
Deploy WASM modules at runtime without recompiling
Evaluate the full PolicyContext including trust tier, channel, and mesh state

Cannot do

Override constitutional rules — a WASM Allow cannot override a Tenet Block
Remove existing rules from the engine — there is no removal API
Modify evaluation order — constitutional rules always evaluate first
Disable the audit trail — SovereigntyRule blocks this unconditionally
Downgrade a severity — Block cannot become Warn through any mechanism
Escape the WASM sandbox — modules cannot access host filesystem or network
For Integrators

Two paths to inject constraints

Path 1 — Compile Time

Native Rust Rules

Implement the PolicyRule trait and register with the engine. Full access to the Rust type system. Appropriate for tightly coupled integrations.

impl PolicyRule for CognitiveWisdomRule { fn name(&self) -> &str { "CognitiveWisdom" } fn evaluate(&self, ctx: &PolicyContext) -> Option<PolicyDecision> { // Reason about whether the // action is wise, not just // whether it's permitted. } }
Path 2 — Runtime

WASM Policy Modules

Deploy sandboxed WebAssembly modules without recompiling ZeroPoint. Receive PolicyContext as JSON, return PolicyDecision as JSON. Memory-isolated, fuel-limited.

// Load at runtime let registry = PolicyModuleRegistry::new()?; registry.load(wasm_bytes)?; engine.set_wasm_registry(registry); // Module receives full context // Module returns graduated decision // Sandbox prevents host access
Trust Distribution

Cryptographic keys below the policy engine

ZeroPoint solves the key distribution problem with a three-level hierarchy that exists below the policy engine. The mechanism is primitive — verification is deterministic and requires no network, no policy state, and no runtime configuration. The decision to delegate is policy-gated; the mechanism of delegation is unconditional.

Each level holds an Ed25519 keypair and a signed certificate attesting its role. Certificates form a chain — any node can verify an agent's identity by walking the chain back to a known genesis key, with no external dependencies.

zp-keys

Primitive, Not Policy

The key hierarchy has no dependency on the policy engine. You need keys to establish the engine's authority across nodes — so keys cannot depend on the engine existing. This avoids a circular dependency.

six-invariants

Chain Verification

Every chain is verified against six invariants: valid signatures, issuer linkage, role hierarchy, monotonic depth, no expired certificates, and correct hash linking.

policy-gated

Governed Delegation

Issuing a child certificate flows through the policy engine as ActionType::KeyDelegation (Critical risk). The mechanism signs; the policy decides when.

portable

Portable Chains

Agent certificate chains are self-contained — a single JSON file carries genesis, operator, and agent certificates. A remote node verifies the chain with only the genesis public key.

Peer Trust

Governed introduction between nodes

When two ZeroPoint nodes meet, the introduction protocol establishes trust. The protocol uses zp-keys for chain verification and generates PolicyContext events for the policy engine to evaluate.

Same Genesis

Nodes from the same deployment. High risk. Both share a genesis key, so the chain is a known trust domain.

Cross Genesis

Nodes from different deployments. Critical risk. Different trust roots require operator-configured policy to allow.

Guarantees

Architectural properties that hold under all conditions

additive-only

Constraints Only Tighten

There is no mechanism to loosen, override, or remove a constraint once loaded. Rules can only be added, never subtracted.

constitutional-supremacy

Constitutional Supremacy

HarmPrincipleRule and SovereigntyRule are compiled into the engine, loaded first, and cannot be bypassed by any runtime mechanism.

deterministic

Deterministic Evaluation

Given the same PolicyContext and rule set, the engine always produces the same decision. No randomness, no external state dependencies.

sandbox-isolation

Sandbox Isolation

WASM policy modules execute in memory-isolated sandboxes. A malicious module can at most return an incorrect decision. It cannot escape.

audit-complete

Audit Completeness

Every evaluation is logged with the contributing rule name and decision. The receipt chain records which rules participated in every decision.

monotonic-severity

Monotonic Severity

The engine selects max(severity) across all decisions. Block > Review > Warn > Sanitize > Allow. This ordering is hardcoded.

Wisdom yields to principle.
Never the other way around.

An external cognitive layer can reason about whether an action is wise.
ZeroPoint's constitutional rules determine whether it's permitted.
If they disagree, principle wins. The receipt records both assessments.
