# Capability Verification Receipts

**Status**: Design Sketch
**Author**: Ken Romero + Claude
**Date**: 2026-03-31
**Context**: The Ember `clientBaseUrl` bug — ZP correctly wrote API keys to `.env` and emitted `tool:configured`, but Ember's route handlers silently bypassed server-side key resolution. The existing receipt chain had no mechanism to detect this class of failure.

---

## Problem Statement

ZP's current receipt chain validates the **supply side** of credential delivery:

```
tool:configured:ember        — .env exists with "Generated by: zp configure" marker
tool:preflight:passed:ember  — Docker/compose/port checks green
tool:launched:ember           — process spawned on port 9102
```

But it is blind to the **consumption side** — whether the tool's runtime actually resolves the configured credentials into working API calls. Three failure classes go undetected:

1. **Code-path bypass**: Tool has a conditional branch that skips reading `.env` (the Ember bug)
2. **Provider mapping gap**: Tool's internal provider registry doesn't map the env var ZP wrote
3. **Key staleness**: API key was valid at configure time but has since been revoked or rotated

These are all runtime failures that happen *after* a correct configure, *after* a passing preflight, and *after* a successful launch — in the gap between "keys delivered" and "keys used."

---

## Proposed Receipt Tiers

### Tier 0: `tool:configured` (exists today)

**When**: After `zp configure auto` writes `.env`
**Assertion**: The `.env` file exists and contains the "Generated by: zp configure" marker
**Proves**: ZP delivered credentials to the tool's filesystem

```
tool:configured:ember → detail: "launch_method=npm"
```

### Tier 1: `tool:providers:resolved` (new)

**When**: After tool launch, once health is `up`
**Assertion**: The tool's runtime loaded the credentials ZP configured
**Proves**: The tool's provider registry sees the keys (not that they work, just that they're loaded)

```
tool:providers:resolved:ember → detail: "openai=loaded,anthropic=loaded,google=missing"
```

**Mechanism**: ZP queries the tool's provider introspection endpoint (declared in manifest). The endpoint returns which providers have non-empty API keys at runtime — never the keys themselves.

### Tier 2: `tool:capability:verified` (new)

**When**: After providers are resolved, ZP sends a lightweight verify probe
**Assertion**: The configured credential actually authenticates with the upstream provider
**Proves**: End-to-end credential validity — from ZP vault → `.env` → tool runtime → provider API

```
tool:capability:verified:ember:reasoning_llm → detail: "provider=openai,model=gpt-4o-mini,latency_ms=340"
tool:capability:verified:ember:tts           → detail: "provider=openai-tts,latency_ms=210"
tool:capability:degraded:ember:image_gen     → detail: "provider=seedream,error=401 Unauthorized"
```

**Mechanism**: ZP calls the tool's capability verify endpoint with a minimal dry-run request. The tool makes a real (but cheap) API call to confirm the key works.

### Tier 3: `tool:capability:live` (future — derived from proxy traffic)

**When**: Ongoing, during normal operation
**Assertion**: The capability is actively serving real requests
**Proves**: Continuous operational health, not just point-in-time verification

```
tool:capability:live:ember:reasoning_llm → detail: "requests_1h=47,errors_1h=0,p50_ms=1200"
```

**Mechanism**: Already partially implemented via `tool:traffic:request` and `tool:traffic:error` receipts. This tier aggregates those into per-capability health signals.

## Manifest Extension

Add an optional `[verification]` section to `.zp-configure.toml`:

```toml
[tool]
name = "ember"
version = "0.6.0"
description = "AI learning environment"

# ... existing required/optional capabilities ...

[verification]
# Endpoint that returns which providers loaded successfully at runtime.
# Response: { "providers": { "openai": { "loaded": true }, "anthropic": { "loaded": true } } }
providers_endpoint = "/api/server-providers"

# Per-capability verify endpoints.
# ZP sends a GET (or POST with empty/minimal body) and checks for 2xx.
[verification.endpoints]
reasoning_llm  = "/api/verify-model"
tts            = "/api/health"          # Some tools only have a general health check
image_gen      = "/api/verify-image-provider"
video_gen      = "/api/verify-video-provider"
pdf            = "/api/verify-pdf-provider"
image_analysis = "/api/analyze-images"  # v0.6.0 — vision-based pedagogical image analysis (SSE)

# How to probe each endpoint. Default is GET.
# For endpoints that need parameters, declare them here.
[verification.probes]
reasoning_llm.method = "GET"
reasoning_llm.headers = { "x-model" = "openai/gpt-4o-mini" }
image_gen.method = "GET"
image_gen.headers = { "x-provider" = "seedream" }
```

### Design Principles

1. **Optional and incremental**: Tools without `[verification]` still get Tier 0 receipts. Adding `providers_endpoint` enables Tier 1. Adding `endpoints` enables Tier 2.

2. **Tool declares, ZP probes**: ZP doesn't need to understand the tool's internal API. The tool declares what to call; ZP just makes the call and interprets HTTP status.

3. **No key exposure**: The `providers_endpoint` returns loaded/not-loaded booleans, never the actual keys. ZP already knows what it configured — it's just confirming the tool received them.

4. **Cheap probes**: Verify endpoints should be lightweight. For LLMs, a model-list call (not a completion). For TTS, a voices-list call. For image gen, a connectivity check.

### Extended `ToolChainState` in `tool_chain.rs`

```rust
#[derive(Debug, Clone, Serialize)]
pub struct ToolChainState {
    pub name: String,
    // Tier 0
    pub configured: bool,
    pub configured_at: Option<String>,
    // Preflight
    pub preflight_passed: bool,

    pub capabilities: Vec<CapabilityState>,
    // Derived
    pub ready: bool,
    pub verified: bool,  // ready + all required capabilities verified
}

#[derive(Debug, Clone, Serialize)]
pub struct CapabilityState {
    pub capability: String,
    pub status: CapabilityStatus,
    pub verified_at: Option<String>,
    pub provider: Option<String>,
    pub detail: Option<String>,
}

#[derive(Debug, Clone, Serialize)]
pub enum CapabilityStatus {
    Pending,      // Not yet probed
    Verified,     // Probe returned 2xx
    Degraded,     // Probe returned non-2xx but tool is functional (optional capability)
    Failed,       // Probe returned non-2xx on a required capability
}
```

### Verification runner in `onboard/verify.rs` (new file)

```rust
use crate::tool_chain::{emit_tool_receipt, ToolEvent};
use crate::tool_state::events as te;
use zp_engine::capability::ToolManifest;
use std::sync::Mutex;
use zp_audit::store::AuditStore;

/// Verification config parsed from .zp-configure.toml [verification] section.
#[derive(Debug, Clone, Deserialize)]
pub struct VerificationConfig {
    pub providers_endpoint: Option<String>,
    pub endpoints: HashMap<String, String>,
    pub probes: HashMap<String, ProbeConfig>,
}

#[derive(Debug, Clone, Deserialize)]
pub struct ProbeConfig {
    pub method: Option<String>,    // GET or POST, default GET
    pub headers: Option<HashMap<String, String>>,
    pub body: Option<String>,      // JSON body for POST probes
}

/// Run Tier 1 + Tier 2 verification after tool launch.
///
/// Called from the health-up transition in the tool state machine.
pub async fn verify_tool_capabilities(
    tool_name: &str,
    tool_port: u16,
    manifest: &ToolManifest,
    verification: &VerificationConfig,
    audit_store: &Mutex<AuditStore>,
) -> VerificationResult {
    let base = format!("http://localhost:{}", tool_port);
    let client = reqwest::Client::builder()
        .timeout(std::time::Duration::from_secs(10))
        .build()
        .unwrap();

    let mut result = VerificationResult::default();

    // ── Tier 1: Provider resolution check ──────────────────────────
    if let Some(providers_ep) = &verification.providers_endpoint {
        let url = format!("{}{}", base, providers_ep);
        match client.get(&url).send().await {
            Ok(resp) if resp.status().is_success() => {
                // Parse response to determine which providers loaded
                if let Ok(body) = resp.json::<serde_json::Value>().await {
                    let mut loaded = Vec::new();
                    let mut missing = Vec::new();

                    if let Some(providers) = body.as_object() {
                        for (id, info) in providers {
                            // server-providers returns { "openai": { "models": [...] } }
                            // A provider with any content means it loaded
                            if info.as_object().map_or(false, |o| !o.is_empty()) {
                                loaded.push(id.clone());
                            } else {
                                missing.push(id.clone());
                            }
                        }
                    }

                    let detail = format!(
                        "loaded={},missing={}",
                        loaded.join("+"),
                        if missing.is_empty() { "none".into() } else { missing.join("+") }
                    );

                    emit_tool_receipt(
                        audit_store,
                        &te::for_tool(te::PROVIDERS_RESOLVED, tool_name),
                        Some(&detail),
                    );

                    result.providers_resolved = true;
                    result.loaded_providers = loaded;
                    result.missing_providers = missing;
                }
            }
            Ok(resp) => {
                // Non-2xx — tool's endpoint exists but returned an error
                let detail = format!("status={}", resp.status().as_u16());
                emit_tool_receipt(
                    audit_store,
                    &te::for_tool(te::PROVIDERS_RESOLVED, tool_name),
                    Some(&format!("error: {}", detail)),
                );
            }
            Err(e) => {
                // Connection error — endpoint doesn't exist or tool isn't ready
                let detail = format!("error={}", e);
                emit_tool_receipt(
                    audit_store,
                    &te::for_tool(te::PROVIDERS_RESOLVED, tool_name),
                    Some(&format!("error: {}", detail)),
                );
            }
        }
    }

    // ── Tier 2: Per-capability verification ─────────────────────────
    for (capability, endpoint) in &verification.endpoints {
        let url = format!("{}{}", base, endpoint);
        let probe = verification.probes.get(capability);
        let method = probe
            .and_then(|p| p.method.as_deref())
            .unwrap_or("GET");

        let mut req = match method {
            "POST" => client.post(&url),
            _ => client.get(&url),
        };

        // Add probe-specific headers
        if let Some(headers) = probe.and_then(|p| p.headers.as_ref()) {
            for (k, v) in headers {
                req = req.header(k.as_str(), v.as_str());
            }
        }

        // Add probe body if POST
        if let Some(body) = probe.and_then(|p| p.body.as_ref()) {
            req = req.body(body.clone());
        }

        let is_required = manifest
            .required
            .iter()
            .any(|r| r.capability == *capability);

        match req.send().await {
            Ok(resp) if resp.status().is_success() => {
                let detail = format!("status={},latency_ms=TODO", resp.status().as_u16());
                let event = te::capability(te::CAPABILITY_VERIFIED, tool_name, capability);
                emit_tool_receipt(audit_store, &event, Some(&detail));

                result.capabilities.push(CapabilityResult {
                    capability: capability.clone(),
                    status: "verified".into(),
                    detail,
                });
            }
            Ok(resp) => {
                let status = resp.status().as_u16();
                let body_text = resp.text().await.unwrap_or_default();
                let detail = format!("status={},body={}", status, &body_text[..body_text.len().min(200)]);

                let event_type = if is_required {
                    te::CAPABILITY_FAILED
                } else {
                    te::CAPABILITY_DEGRADED
                };
                let event = te::capability(event_type, tool_name, capability);
                emit_tool_receipt(audit_store, &event, Some(&detail));

                result.capabilities.push(CapabilityResult {
                    capability: capability.clone(),
                    status: if is_required { "failed" } else { "degraded" }.into(),
                    detail,
                });
            }
            Err(e) => {
                let detail = format!("error={}", e);
                let event_type = if is_required {
                    te::CAPABILITY_FAILED
                } else {
                    te::CAPABILITY_DEGRADED
                };
                let event = te::capability(event_type, tool_name, capability);
                emit_tool_receipt(audit_store, &event, Some(&detail));

                result.capabilities.push(CapabilityResult {
                    capability: capability.clone(),
                    status: if is_required { "failed" } else { "degraded" }.into(),
                    detail,
                });
            }
        }
    }

    result
}
```

### Trigger point: health-up transition

The verification runs once, triggered by the first `tool:health:up` receipt after launch. This ensures the tool is actually accepting connections before probing.

In `lib.rs` (or a dedicated health monitor), when the proxy observes a successful response from a tool that was previously in `Starting` state:

```rust
// After emitting tool:health:up receipt...
if let Some(manifest) = load_manifest(&tool_path) {
    if let Some(verification) = manifest.verification {
        tokio::spawn(async move {
            verify_tool_capabilities(
                &tool_name,
                tool_port,
                &manifest,
                &verification,
                &audit_store,
            ).await;
        });
    }
}
```

## Chain State Derivation

The readiness computation in `query_tool_readiness()` extends naturally:

```rust
// Current
state.ready = state.configured && state.preflight_passed;

// Extended
state.verified = state.ready
    && state.providers_resolved
    && state.capabilities.iter()
        .filter(|c| is_required_capability(c.capability))
        .all(|c| c.status == CapabilityStatus::Verified);
```

The `verified` field gives the dashboard a stronger signal than `ready`:

| Badge | Meaning |
|-------|---------|
| `configured` | ZP wrote the `.env` |
| `ready` | Configured + preflight passed |
| `verified` | Ready + runtime confirmed all required capabilities work |

## Receipt-Driven Configuration

### Problem

Tools like Ember have feature toggles (image generation, video generation, TTS provider selection, model preferences) that are normally controlled through a browser-side settings UI requiring direct API key entry. Under ZP governance, API keys are vaulted and never exposed to the tool's frontend — so the settings UI is effectively dead.

The receipt chain already proves capability state. Extending it to **carry configuration state** means:

1. Configuration changes are auditable (they're receipts in the chain)
2. ZP is the single authority for what's enabled and how
3. No API keys touch Ember's browser-side settings store
4. The tool doesn't need a separate "managed mode" settings backend — it just reads receipts

### New Receipt: `tool:capability:configured`

**When**: After ZP resolves which capabilities should be active and with what parameters
**Assertion**: ZP has decided how this capability should be configured at the tool
**Proves**: Configuration intent — what ZP wants the tool to do (distinct from verification, which proves it *can* do it)

```
tool:capability:configured:ember:image_gen → {
  "enabled": true,
  "provider": "nano_banana",
  "model": "gemini-2.5-flash-preview-image-generation",
  "parameters": {
    "aspect_ratio": "16:9",
    "max_concurrent": 3
  }
}

tool:capability:configured:ember:reasoning_llm → {
  "enabled": true,
  "provider": "anthropic",
  "model": "claude-sonnet-4-6",
  "parameters": {
    "max_tokens": 8192,
    "temperature": 0.7
  }
}

tool:capability:configured:ember:tts → {
  "enabled": true,
  "provider": "openai-tts",
  "model": "tts-1",
  "parameters": {
    "voice": "nova",
    "speed": 1.0
  }
}
```

This receipt is emitted **before** launch — it represents ZP's intent. The Tier 2 `tool:capability:verified` receipt then confirms the configuration actually works.


### Ember-Side Integration

Ember adds a thin adapter that reads configuration receipts instead of browser localStorage when ZP-managed:

```typescript
// lib/settings/zp-config-adapter.ts

interface ZPConfigReceipt {
  capability: string;
  enabled: boolean;
  provider: string;
  model: string;
  parameters: Record<string, unknown>;
  receipt_id: string;
  issued_at: string;
}

/**
 * Fetch configuration receipts from ZP's local sidecar.
 * Returns null if not running under ZP (endpoint doesn't exist).
 */
async function fetchZPConfigReceipts(): Promise<ZPConfigReceipt[] | null> {
  try {
    const zpPort = process.env.ZP_SIDECAR_PORT;
    if (!zpPort) return null;

    const resp = await fetch(
      `http://localhost:${zpPort}/zp/receipts/configured/${process.env.ZP_TOOL_NAME}`
    );
    if (!resp.ok) return null;

    const data = await resp.json();
    return data.receipts;
  } catch {
    return null;  // Not ZP-managed, fall back to normal settings
  }
}

/**
 * Map ZP configuration receipts to Ember's settings shape.
 * Called on server startup and exposed via /api/settings/managed.
 */
function receiptToSettings(receipts: ZPConfigReceipt[]): Partial<EmberSettings> {
  const settings: Partial<EmberSettings> = {};

  for (const r of receipts) {
    switch (r.capability) {
      case 'image_gen':
        settings.imageGenerationEnabled = r.enabled;
        settings.imageProviderId = r.provider;
        settings.imageModelId = r.model;
        break;
      case 'video_gen':
        settings.videoGenerationEnabled = r.enabled;
        settings.videoProviderId = r.provider;
        settings.videoModelId = r.model;
        break;
      case 'reasoning_llm':
        settings.reasoningProviderId = r.provider;
        settings.reasoningModelId = r.model;
        break;
      case 'tts':
        settings.ttsProviderId = r.provider;
        settings.ttsModelId = r.model;
        if (r.parameters.voice) settings.ttsVoice = r.parameters.voice as string;
        break;
    }
  }

  return settings;
}
```

### Reconfiguration Flow

When a user wants to change settings (e.g., switch image provider):

```
User runs: zp configure ember --set image_gen.provider=seedream

ZP:
  1. Validates the new configuration against available vault keys
  2. Emits: tool:capability:configured:ember:image_gen (new receipt, new provider)
  3. Updates .env if needed (new key injection)
  4. Sends SIGHUP or calls /api/settings/reload on Ember
  5. Re-runs Tier 2 verification for image_gen
  6. Emits: tool:capability:verified:ember:image_gen (confirms new config works)

Ember:
  1. Receives reload signal
  2. Re-fetches /zp/receipts/configured/ember
  3. Updates settings store with new receipt data
  4. Settings UI reflects the change with the new receipt ID
```

The entire change is captured in the receipt chain:

```
r_abc123  tool:capability:configured:ember:image_gen  provider=nano_banana     10:30 AM
r_xyz999  tool:capability:configured:ember:image_gen  provider=seedream        2:15 PM
r_xyz000  tool:capability:verified:ember:image_gen    provider=seedream,ok     2:15 PM
```

### Manifest Extension

Add configurable parameters to `.zp-configure.toml` capability declarations:

```toml
[[required]]
capability = "reasoning_llm"
env_vars = ["ANTHROPIC_API_KEY"]
configurable = ["provider", "model", "temperature", "max_tokens"]
defaults = { provider = "anthropic", model = "claude-sonnet-4-6" }

[[optional]]
capability = "image_gen"
env_vars = ["IMAGE_NANO_BANANA_API_KEY"]
configurable = ["provider", "model", "aspect_ratio", "max_concurrent"]
defaults = { provider = "nano_banana", model = "gemini-2.5-flash-preview-image-generation", enabled = true }
notes = "AI image generation for slides and whiteboard illustrations"

[[optional]]
capability = "tts"
env_vars = ["OPENAI_API_KEY"]
configurable = ["provider", "model", "voice", "speed"]
defaults = { provider = "openai-tts", model = "tts-1", voice = "nova" }
```

The `configurable` array tells ZP which parameters users can adjust via `zp configure`. The `defaults` provide sane starting values. Both feed into the configuration receipts.

## Migration Path

1. **Phase 1**: Add `[verification]` parsing to `ToolManifest` (backward compatible — field is optional)
2. **Phase 2**: Implement `verify_tool_capabilities()` in a new `onboard/verify.rs`
3. **Phase 3**: Wire into health-up transition
4. **Phase 4**: Extend `query_tool_readiness()` and dashboard rendering
5. **Phase 5**: Add `verified` as a governance gate (optional — tools can run without verification, but agents see the degraded badge)
6. **Phase 6**: Implement `tool:capability:configured` receipts with `configurable` + `defaults` in manifest
7. **Phase 7**: Add ZP sidecar endpoint `/zp/receipts/configured/{tool}` for tool-side consumption
8. **Phase 8**: Ember-side: `zp-config-adapter.ts` + `/api/settings/managed` endpoint + managed-mode settings UI
9. **Phase 9**: `zp configure {tool} --set` CLI for runtime reconfiguration with receipt chain audit trail

| `configure.rs` | Writes `.env` from vault | Also emits `tool:capability:configured` receipts from manifest defaults |
| `preflight.rs` | Checks infra (Docker, compose, ports) | No change |
| `tool_chain.rs` | Derives readiness from chain | Also derives verification + configuration state |
| `tool_state.rs` | Event taxonomy | New capability + configuration events |
| `lib.rs` (launch) | Spawns process, emits launch receipt | Triggers verification after health-up |
| Proxy | Emits traffic receipts | Injects capability headers derived from configuration receipts |
| Sidecar (new) | — | Serves `/zp/receipts/configured/{tool}` for tool-side consumption |
| Tool settings (new) | Browser localStorage | Reads from ZP receipts when managed; falls back to localStorage otherwise |

The key insight is that this slots into the existing state-derivation-from-chain architecture cleanly. No cached state, no side channels — just new receipt types that `query_tool_readiness()` learns to interpret. Configuration receipts extend this naturally: the receipt chain becomes the **single source of truth** for both "what should run" and "is it running correctly."
